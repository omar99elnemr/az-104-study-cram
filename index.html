Administer Identity

01

Microsoft Entra ID — Core Concepts

  • Entra ID (formerly Azure AD) — cloud-based identity & access management service
  • Identity — any object that can be authenticated (user, group, managed identity, service principal)
  • Account = Identity + data attributes (department, phone, manager, etc.)
  • Tenant/Directory — dedicated Entra ID instance created when you sign up for any Microsoft cloud service. Each Azure subscription maps to one tenant.
  • Managed Identity — identity for Azure services to authenticate to other Azure services. No credential management needed. System-assigned (tied to resource lifecycle) or User-assigned (independent lifecycle, reusable).
  • Service Principal — like a service account; used by apps & automated processes

Entra ID vs. AD DS

  • Entra ID: HTTP/HTTPS, SAML/OpenID/OAuth, cloud-managed
  • AD DS: LDAP, Kerberos, on-premises servers required
  • Entra ID: Federation with Facebook/Google possible
  • Hybrid: use Entra ID Connect to sync on-prem to cloud
  • Entra ID Connect Health — monitors sync health

Entra ID Editions

  • Free — SSO, self-service password change, basic user mgmt
  • P1 — Conditional Access, Advanced Group Mgmt, MFA, SSPR, Hybrid Entra Join
  • P2 — Identity Protection, Privileged Identity Mgmt (PIM), risk-based Conditional Access
  • Governance — PIM + Identity Lifecycle, access reviews, auto provisioning

Device Identities

  • Entra Registered — personal BYOD devices; user signs in with personal account
  • Entra Joined — cloud-only join; org-owned; no on-prem AD needed; Windows 10/11
  • Hybrid Entra Joined — joined to both on-prem AD DS and Entra ID
  • Device management via Microsoft Intune (MDM/MAM)

User Accounts

  • Cloud Identity — created directly in Entra ID
  • Directory-Synced (Hybrid) — synced from on-prem AD via Entra ID Connect
  • Guest User (B2B) — external users invited via Azure AD B2B; use their own credentials
  • Deleted users go to Recycle Bin for 30 days before permanent deletion
  • Bulk operations — create/invite/delete users via CSV upload in portal
  • B2C — consumer-facing identity (NOT B2B); for customers of your apps

Group Accounts

  • Security Group — manage access to resources; can assign licenses
  • Microsoft 365 Group — collaboration (Teams, SharePoint); includes mailbox
  • Assigned — manually add/remove members
  • Dynamic User — auto-membership based on user attributes — requires P1/P2
  • Dynamic Device — auto-membership based on device attributes
  • Nested groups — a group can be a member of another group

Self-Service Password Reset (SSPR)

  • Allows users to reset/unlock passwords without IT helpdesk
  • Requires P1 or P2 (free for Global Admins)
  • Auth methods: email, mobile app, security questions, phone call, SMS
  • Can require 1 or 2 auth methods
  • Password Writeback — syncs cloud password changes back to on-prem AD (needs Entra ID Connect + P1)
  • SSPR scope: None / Selected (a group) / All users

Conditional Access HIGH WEIGHT

  • Policy engine: If [signals] → Then [controls]
  • Requires Entra ID P1 or P2
  • Signals (conditions): User/group, Cloud app, Location (IP), Device platform, Device state (compliant/hybrid-joined), Sign-in risk (P2), User risk (P2)
  • Controls (grant/session): Block access, Require MFA, Require compliant device, Require Hybrid Entra join, Require approved app, Require password change
  • Named Locations — define trusted IP ranges or countries; use in location conditions
  • Report-only mode — evaluate policy without enforcing (test before enabling)
  • What If tool — simulate which policies would apply to a specific user/scenario
  • Emergency "break glass" accounts — excluded from CA policies; use for emergency admin access
  • CA policies are evaluated after first-factor authentication; they don't replace it
⚠ Exam trap: Conditional Access does NOT block sign-in before first authentication. It steps in after credentials are verified to enforce further controls (MFA, compliant device, etc.).

Privileged Identity Management (PIM) HIGH WEIGHT

  • Requires Entra ID P2
  • Provides just-in-time privileged access — activate roles only when needed
  • Eligible assignment — user must activate the role (time-limited, requires justification/approval/MFA)
  • Active assignment — role is always active (permanent or time-bound)
  • Activation — eligible user requests activation; can require approval, MFA, justification, ticket number
  • Access Reviews — periodic review of who has privileged roles; auto-remove stale assignments
  • PIM covers both Entra ID roles (Global Admin, etc.) and Azure resource roles (Owner, Contributor)
  • Alerts — PIM notifies when roles are assigned outside PIM, or when too many Global Admins exist
  • Audit history — tracks all role activations and assignments
Exam tip: PIM = P2. Eligible = activate on demand. Active = always on. Access Reviews let you automatically revoke roles that weren't reconfirmed.

Multi-Tenant & Federation

  • Each tenant is isolated by default
  • B2B Collaboration — invite external users as guests into your tenant
  • Cross-tenant synchronization — sync users across tenants in same org
  • A user can be a member of one tenant but a guest in multiple others
  • Subscriptions belong to exactly one tenant; you can transfer a subscription to another tenant (billing transfer)
  • Federation — trust external identity providers (ADFS, Google, etc.) via SAML/WS-Fed
Exam tip: SSPR = P1/P2. Dynamic groups = P1/P2. PIM = P2. Conditional Access = P1/P2. Password writeback = Entra Connect + P1. Guest = B2B (not B2C). B2C is for consumer apps.

Administer Governance & Compliance

02

Azure Hierarchy (Top → Bottom)

  • Management Groups → Subscriptions → Resource Groups → Resources
  • Policies and RBAC assigned at higher levels inherit downward
  • Root Management Group at top; max 6 levels of management groups (not counting root)
  • One subscription can only belong to one management group at a time
  • Up to 10,000 management groups in a single directory

Subscriptions

  • Billing boundary and access control boundary
  • Each subscription belongs to exactly one tenant
  • Types: Pay-as-you-go, Enterprise Agreement (EA), CSP, Free trial, Visual Studio
  • Resource limits (quotas) are per subscription — request increases if needed

Resource Groups

  • Logical container for resources sharing the same lifecycle
  • Resources in only one RG at a time, but can span regions
  • Deleting an RG deletes ALL resources inside it
  • RBAC and Policy applied at RG level apply to all resources within

Azure Resource Tags

  • Key-value pairs for organizing/filtering/billing (e.g., Environment=Production)
  • Tags are NOT inherited from RG or subscription by default
  • Max 50 tags per resource; max 512 characters for key, 256 for value
  • Use Azure Policy to enforce tagging (Deny if missing, or Append missing tag)
  • Tags appear on invoices for cost allocation
  • Use Modify policy effect (with managed identity) to add/change tags on existing resources

Azure Resource Locks

  • CanNotDelete (Delete lock) — users can read/modify but NOT delete
  • ReadOnly lock — users can read but NOT modify or delete (similar to Reader role)
  • Locks override RBAC — even Owners are blocked by locks
  • Locks are inherited by child resources
  • Must remove the lock before you can perform the blocked action
  • Lock scope: subscription, RG, or individual resource
⚠ Exam: ReadOnly lock on storage account prevents listing keys (because key listing is a write operation on the resource metadata). Owner role can still be blocked by a lock.

Azure Policy

  • Enforces organizational standards and assesses compliance at scale
  • Policy Definition — single rule (e.g., "VMs must be in specific regions")
  • Initiative (Policy Set) — grouped collection of policy definitions
  • Policy Assignment — applying to a scope (MG, subscription, RG)
  • Effects (in order of evaluation): Disabled → Append → Deny → Audit → Modify → AuditIfNotExists → DeployIfNotExists
  • Deny — blocks non-compliant resource creation/update
  • Audit — logs compliance issues but doesn't block
  • Append — adds fields to a resource (e.g., add a tag)
  • Modify — changes properties (requires managed identity for remediation)
  • DeployIfNotExists — deploys a related resource if missing (e.g., deploy diagnostic settings)
  • Existing non-compliant resources get flagged; need a remediation task to fix existing resources
  • Exclusions — exclude specific scopes or resources from a policy assignment

Role-Based Access Control (RBAC)

  • Security Principal (who) + Role Definition (what) + Scope (where) = Role Assignment
  • Built-in roles: Owner (full + manage access), Contributor (full, NO manage access), Reader (read-only), User Access Administrator (manage access only)
  • RBAC is additive — most permissive wins (unless deny assignment blocks it)
  • Scope hierarchy: Management Group → Subscription → RG → Resource (assignments inherit downward)
  • Custom roles — create your own with specific actions/notActions; scoped to MG or subscription
  • Deny assignments — block specific principals regardless of role; used by Azure Blueprints/Managed Apps
  • Check access: Access Control (IAM) blade on any resource → Check access tab

Moving Resources OFTEN TESTED

  • Resources can be moved between resource groups, subscriptions, or regions
  • During a move, both source and destination RGs are locked (no modifications during move)
  • Not all resources support move — check move support for each resource type before exam
  • Moving a VM also moves its disks and NICs; NSG and VNet stay in source (must be in same VNet)
  • Moving between subscriptions — destination subscription must be in same Entra ID tenant, or transfer billing separately
  • RBAC role assignments do NOT move — must reassign roles in destination
  • Azure Policy assignments do NOT move — must reassign policy in destination
  • Resource IDs change when moved to a different subscription
  • Moving between regions — use Azure Resource Mover tool; resources are deployed fresh in new region
⚠ Exam: RBAC and policy assignments must be re-created after a subscription move. Also: Key Vault can be moved but certificate references may break.

Cost Management + Azure Advisor NEW

  • Azure Cost Management — analyze, monitor, and optimize Azure spend
  • Budgets — set spending thresholds; triggers alerts at % of budget (e.g., 80%, 100%); can trigger an action (email, webhook, action group)
  • Budgets do NOT automatically stop resources — they only alert
  • Cost Alerts: Budget alerts, Credit alerts (EA only), Department spending quota alerts
  • Cost Analysis — visualize spending by resource, service, tag, location, time
  • Reservations — 1 or 3-year commitment for VMs, SQL, etc.; save up to 72% vs pay-as-you-go
  • Azure Hybrid Benefit — use on-prem Windows Server / SQL Server licenses in Azure (save ~40%)
  • Spending limits — available for some subscription types (Visual Studio, Free Trial); hard cap that stops resources when reached
Exam tip: Budgets = alerts only, do NOT stop resources. Use Spending Limit (Visual Studio/Free subs) to hard-stop. Tags enable cost allocation by team/project.

Administer Azure Resources

03

Admin Tools

  • Azure Portal — GUI; best for ad-hoc tasks
  • Azure CLI — cross-platform; az <command>
  • Azure PowerShell — Get-Az*, New-Az*, Set-Az*, Remove-Az*
  • Azure Cloud Shell — browser-based CLI/PS; no local install
  • Azure REST API — programmatic access; all tools use this underneath

Azure Resource Manager (ARM)

  • Management layer for ALL Azure resources — portal, CLI, APIs all go through ARM
  • Handles authentication via Entra ID
  • Provides RBAC, locks, tags, resource grouping
  • Declarative deployment via templates (idempotent)
  • Resources in a template deployed as a single unit

ARM Templates

  • JSON files that declaratively define infrastructure — Infrastructure as Code
  • Sections: $schema, contentVersion, parameters, variables, resources, outputs
  • parameters — inputs at deployment time (reusable templates)
  • variables — computed values used within the template
  • resources — array of resources to deploy (mandatory)
  • outputs — return values after deployment (e.g., connection string)
  • Supports linked/nested templates for modular design
  • Idempotent — deploying same template twice = same result
  • Deployment modes: Incremental (adds/updates, keeps existing resources) vs Complete (deletes resources not in template)

Azure Bicep

  • DSL that compiles (transpiles) to ARM JSON — cleaner syntax, same capabilities
  • Supports modules, conditions, loops natively
  • Idempotent like ARM; preferred for new IaC deployments on Azure
  • az bicep build — converts .bicep to ARM JSON
  • az deployment group create --template-file main.bicep — deploy directly

Azure Key Vault HIGH WEIGHT

  • Securely store and access: Secrets (passwords, connection strings), Keys (encryption keys), Certificates (SSL/TLS certs)
  • Access control: Vault Access Policy (legacy) or Azure RBAC (recommended)
  • Azure RBAC roles for Key Vault: Key Vault Administrator, Key Vault Secrets Officer, Key Vault Secrets User, Key Vault Reader
  • Soft Delete — deleted secrets/keys retained for 7–90 days (configurable); enabled by default; cannot be disabled once enabled
  • Purge Protection — prevents permanent deletion during soft-delete retention; required for BYOK (Bring Your Own Key)
  • HSM-backed keys — hardware security module (Premium tier); higher security
  • Managed HSM — single-tenant, FIPS 140-2 Level 3; for highest compliance
  • Access from VMs/Apps using Managed Identity (no credentials in code)
  • Key Vault firewall — restrict access to specific VNets/IPs; enable "Allow trusted Microsoft services"
  • Versioning — all secrets/keys are versioned; old versions still accessible
  • Key Rotation — auto-rotate keys on schedule; integrated with storage accounts and SQL
Best practice: App Service / Function App → enable Managed Identity → grant Managed Identity "Key Vault Secrets User" role → reference secrets as @Microsoft.KeyVault(SecretUri=...) in app settings. Zero credentials in code.

Administer Virtual Networking

04

Virtual Networks (VNet)

  • Isolated network in Azure; resources communicate privately within a VNet
  • Defined by a CIDR address space (e.g., 10.0.0.0/16)
  • Subnets — subdivide VNet address space; resources deployed into subnets
  • Azure reserves 5 IPs per subnet: .0 (network), .1 (gateway), .2 & .3 (Azure DNS), .255 (broadcast)
  • VNets are region-specific; can span multiple Availability Zones
  • Address spaces between peered VNets must NOT overlap
  • You can add address spaces to a VNet without downtime

Private IP Addresses

  • Dynamic — assigned by DHCP; can change on stop/start (VM)
  • Static — fixed; required for DNS servers, DCs, firewalls, VPN gateways
  • RFC 1918: 10.x.x.x, 172.16–31.x.x, 192.168.x.x

Public IP Addresses

  • Basic SKU — dynamic or static; no zone redundancy; open by default (NSG optional)
  • Standard SKU — static only; zone-redundant; closed by default (must open via NSG)
  • Standard required for: Load Balancer Standard, App Gateway v2, NAT Gateway, AZs

Network Security Groups (NSG)

  • Filters inbound/outbound traffic for a subnet or NIC
  • Security rules: priority (100–4096), source/dest (IP, CIDR, ASG, service tag), port, protocol, Allow/Deny
  • Lower priority number = processed first; first matching rule wins
  • Default rules (cannot delete, but can override with lower priority): AllowVNetInBound (65000), AllowAzureLoadBalancerInBound (65001), DenyAllInBound (65500)
  • NSG on subnet affects all resources; NSG on NIC affects only that VM
  • When on both: inbound — subnet NSG evaluated first; outbound — NIC NSG first
  • Service Tags — named groups of IPs (e.g., Internet, AzureCloud, Storage, VirtualNetwork) — use in rules instead of managing IP ranges
  • Augmented Security Rules — allow multiple ports, IPs, ranges in a single rule

Application Security Groups (ASG)

  • Group VMs logically by role (WebServers, DBServers) — use in NSG rules instead of IPs
  • VMs must be in the same VNet to use the same ASG
  • A NIC can belong to multiple ASGs

NAT Gateway NEW

  • Provides outbound-only internet connectivity for resources in a subnet
  • Replaces the need for outbound rules on a Load Balancer or instance-level public IPs
  • All VMs in the associated subnet share the NAT Gateway's public IP(s) for outbound traffic
  • Supports multiple public IPs or IP prefixes — scale outbound SNAT ports
  • Fully managed, zone-redundant; no inbound traffic (outbound only)
  • Associated at subnet level; overrides LB outbound rules for that subnet
  • SLA: 99.99%; supports TCP and UDP protocols
  • Use when: many VMs need internet egress but no inbound public IP; need predictable outbound IPs; avoid SNAT port exhaustion

Azure DNS

  • Hosts DNS zones in Azure (authoritative DNS); supports A, AAAA, CNAME, MX, NS, PTR, SOA, SRV, TXT
  • Alias records — point to Azure resources (Public IP, Traffic Manager, CDN); auto-update if IP changes
  • Does NOT support domain registration — buy from registrar, delegate NS to Azure DNS
  • Azure-provided DNS: 168.63.129.16 — used automatically within VNets
  • Custom DNS servers — set at VNet level; all VMs in VNet use them

Private DNS Zones

  • Name resolution for resources within VNets; no custom DNS servers needed
  • Link a Private DNS Zone to a VNet via a VNet link
  • Auto-registration — automatically register VM DNS records when enabled on VNet link
  • Max 1 auto-registration link per VNet; but a zone can be linked to multiple VNets for resolution
  • Works across peered VNets if the zone is linked to each VNet

Administer Intersite Connectivity

05

Virtual Network Peering

  • Connects two VNets (same or different regions) over Microsoft backbone — low latency, high bandwidth
  • VNet Peering — same region; Global VNet Peering — cross-region
  • Peering is non-transitive — A↔B and B↔C does NOT give A↔C connectivity
  • Address spaces must NOT overlap
  • Peering must be created on both sides — bidirectional
  • Traffic is NOT encrypted (but stays on Microsoft backbone)
  • Can peer VNets across different subscriptions and tenants

VPN Gateway

  • Sends encrypted traffic between Azure VNet and on-premises (or another VNet) over public internet
  • Deployed into a dedicated subnet: must be named GatewaySubnet exactly; min /27 recommended
  • SKUs: Basic (legacy), VpnGw1–5 (higher = more throughput/connections/IPs)
  • Active-active config — two tunnel endpoints for high availability
  • Site-to-Site (S2S) — on-prem network to Azure; requires on-prem VPN device with public IP and a Local Network Gateway
  • Point-to-Site (P2S) — individual client to Azure VNet; auth: certificates, RADIUS, or Entra ID
  • VNet-to-VNet — connect two Azure VNets via VPN Gateway (uses IPsec/IKE encryption)
  • IKEv1 vs IKEv2 — IKEv2 is recommended; required for P2S with OpenVPN or IKEv2 protocol

ExpressRoute OFTEN TESTED

  • Private dedicated circuit from on-prem to Azure — does NOT traverse public internet
  • Connectivity via a connectivity provider (telco partner)
  • Speeds: 50 Mbps – 10 Gbps (or 100 Gbps with ExpressRoute Direct)
  • Models: Cloud Exchange co-location, Point-to-point Ethernet, Any-to-any (IPVPN)
  • Peering types: Azure Private Peering (VNet resources), Microsoft Peering (M365, Azure PaaS with public IPs)
  • ExpressRoute Global Reach — connect two on-prem sites through Microsoft backbone (via ER circuits)
  • ExpressRoute FastPath — bypass ER gateway for high-throughput scenarios; traffic goes directly to VMs
  • SLA: 99.95%; higher reliability than VPN
  • Traffic is not encrypted by default over ER; add IPsec on top for encryption
  • Used alongside VPN as a failover or for encryption when needed
VPN vs ExpressRoute: VPN = encrypted, over internet, cheaper, lower bandwidth. ExpressRoute = private circuit, no internet, higher reliability, higher bandwidth, more expensive.

Gateway Transit

  • Allows a peered VNet (spoke) to use the VPN/ExpressRoute gateway of the hub VNet
  • Enable Allow gateway transit on the hub peering + Use remote gateways on the spoke peering
  • Cost savings: don't deploy a gateway in every spoke VNet
  • Only one VNet gateway can be used via transit (hub must have the gateway)

User Defined Routes (UDR)

  • Override Azure's default system routes to control traffic flow
  • Defined in a Route Table; associate Route Table to a subnet
  • Next hop types: VirtualNetworkGateway, VirtualAppliance, VNetLocal, Internet, None (black hole)
  • Common use: force all traffic through NVA/firewall (next hop = NVA private IP)
  • Route priority: User-defined routes > BGP routes > System routes

Service Endpoints

  • Extends VNet identity to Azure services over Microsoft backbone
  • Configured on the subnet level
  • Resource still uses its public endpoint — just restricts to VNet traffic
  • Simpler than Private Endpoint; free; no DNS changes needed

Private Endpoint

  • Gives an Azure service a private IP address within your VNet (Private Link)
  • Traffic never leaves your VNet — fully private
  • Works across VNet peering and VPN/ExpressRoute
  • Requires a Private DNS Zone for name resolution
  • More secure: service has no public IP reachability needed
SE vs PE: Service Endpoint = subnet-level access restriction (public IP stays). Private Endpoint = resource gets private IP in your VNet (fully private, works cross-network).

Administer Network Traffic Management

06

Azure Load Balancer (Layer 4)

  • Distributes TCP/UDP traffic; no HTTP awareness (Layer 4 only)
  • Basic SKU — free; no SLA; up to 300 instances; no zone redundancy
  • Standard SKU — 99.99% SLA; up to 1000 instances; zone-redundant; required for AZs
  • Frontend IP — public or private IP receiving traffic
  • Backend Pool — VMs or VMSS instances serving requests
  • Health Probe — HTTP, HTTPS, or TCP; removes unhealthy VMs
  • Load Balancing Rules — maps frontend IP:port → backend pool:port
  • Inbound NAT Rules — forward specific port to a specific VM (e.g., RDP)
  • Outbound Rules — configure SNAT for outbound internet (Standard only)
  • Session persistence: None (5-tuple), Client IP (2-tuple), Client IP+Protocol (3-tuple)

Azure Application Gateway (Layer 7)

  • Web traffic LB; understands HTTP/HTTPS; URL-based routing
  • Components: Frontend IP → Listener → Routing Rule → Backend Pool + HTTP Settings
  • Listener: Basic (one site) or Multi-site (multiple domains on same GW)
  • Routing Rules: Basic (one backend) or Path-based (/api → pool1, /images → pool2)
  • WAF — OWASP rules; protects vs SQL injection, XSS, etc.; prevention or detection mode
  • SSL Termination — decrypts HTTPS at gateway; reduces backend load
  • End-to-end SSL — re-encrypts from gateway to backend
  • Cookie-based session affinity — App Gateway cookie sticks client to same backend
  • Autoscaling — v2 SKU supports autoscale (min/max instance count)
  • Rewrite rules — modify HTTP headers/URLs on the fly

Azure Firewall NEW

  • Managed, stateful network firewall as a service; fully stateful (tracks connections)
  • Deployed in a dedicated subnet (must be named AzureFirewallSubnet, min /26)
  • Central to hub-and-spoke topology; uses UDRs to route all spoke traffic through it
  • Rule types:
    • DNAT rules — translate inbound public IP:port to private IP:port
    • Network rules — allow/deny by IP, port, protocol
    • Application rules — allow/deny outbound by FQDN, URL category
  • Rule priority — lower number processed first; collections have priorities
  • Azure Firewall Premium — adds TLS inspection, IDPS (Intrusion Detection/Prevention), URL filtering, Web categories
  • Threat Intelligence — alert/deny known malicious IPs and FQDNs (built-in, auto-updated)
  • Firewall Policy — centralized policy object (recommended); reusable across multiple firewalls
  • Azure Firewall vs NSG: Firewall = central managed L3–L7 gateway; NSG = subnet/NIC-level traffic filter
Key pattern: Deploy Azure Firewall in hub VNet → create UDR in each spoke subnet with 0.0.0.0/0 → next hop = Firewall private IP → all internet/cross-spoke traffic inspected.

Azure DDoS Protection

  • DDoS Network Protection (formerly Standard) — per-VNet; adaptive tuning; SLA guarantee; real-time attack metrics; cost credits if scaled up due to DDoS
  • DDoS IP Protection — per-public IP; cheaper; subset of Network Protection features
  • DDoS Basic — free; always on; protects Azure infrastructure, not individual resources
  • Combine with WAF (App Gateway or Front Door) for complete application + network protection

Network Watcher

  • IP flow verify — checks if NSG allows/denies a specific flow (5-tuple)
  • Next hop — shows routing decision for source→destination
  • Connection troubleshoot — test TCP/ICMP connectivity between endpoints
  • Connection Monitor — continuous connectivity monitoring with latency/reachability
  • NSG Flow Logs — logs all traffic through NSG (stored in Storage account as JSON); analyze with Traffic Analytics in Log Analytics
  • Packet Capture — captures packets from VM NIC; stores in storage or VM locally
  • VPN Diagnostics — troubleshoot VPN gateway; collect logs into storage
  • Must be enabled per region; auto-enabled when first diagnostics are used

Administer Azure Storage

07

Storage Accounts

  • Container for: Blob (objects/files), Files (SMB/NFS shares), Queues (messaging), Tables (NoSQL)
  • Standard GPv2 — recommended; HDD-backed; supports all services and all tiers
  • Premium Block Blob — SSD; low latency; for block blobs only; no tier support
  • Premium File Share — SSD; for Azure Files only
  • Premium Page Blob — SSD; for page blobs and unmanaged VM disks
  • Name: globally unique, 3–24 chars, lowercase alphanumeric
  • Endpoint: https://<name>.blob.core.windows.net

Blob Access Tiers

  • Hot — frequent access; highest storage cost, lowest access cost; default
  • Cool — infrequent access; lower storage, higher access cost; min retention 30 days
  • Cold — rarely accessed; even lower storage, higher access; min retention 90 days
  • Archive — offline; lowest storage, highest access cost; min retention 180 days; must rehydrate before reading (up to 15 hours standard, or 1 hour high priority)
  • Tier set at account level (default) or per-blob; Archive can only be set at blob level
  • Early deletion fee applies if blob deleted before minimum retention period

Lifecycle Management

  • Auto-transitions blobs between tiers or deletes based on rules
  • Rule conditions: modifiedBefore, accessedBefore, createdBefore + days
  • Example rule: → Cool after 30 days → Cold after 60 → Archive after 180 → Delete after 365
  • Applied at account or container level; runs once daily
  • Works on block blobs; page blobs cannot be tiered to Archive

Blob Versioning & Soft Delete NEW

  • Blob Versioning — every write creates a new version; previous versions retained; each version addressable; enables point-in-time restore
  • Blob Soft Delete — deleted blobs retained for configurable period (1–365 days); appear as deleted state; can undelete; separate from vault soft delete
  • Container Soft Delete — recover accidentally deleted containers (similar concept)
  • Point-in-time Restore — restore block blobs to a state from a specific point in time; requires: versioning, change feed, soft delete all enabled
  • Change Feed — ordered, durable log of all blob create/modify/delete operations; stored as blob in $blobchangefeed container
  • Blob Snapshots — read-only point-in-time copy of a blob; you create them manually; snapshot charged for delta data

Blob Immutability NEW

  • Time-based Retention Policies — WORM (write once, read many); blobs cannot be modified or deleted during retention period; required for compliance (SEC 17a-4, FINRA, CFTC)
  • Legal Hold — holds blobs indefinitely until hold is cleared (no time limit); used during litigation
  • Immutability can be applied at container or blob version level
  • Once a locked policy is set, it CANNOT be shortened — only extended or held

Object Replication NEW

  • Asynchronously copies block blobs between storage accounts (same or different regions/subscriptions)
  • Requires: blob versioning and change feed enabled on both source and destination
  • Use case: geo-distribution, DR, reduce read latency by replicating to regions closer to users
  • Replication is one-way (source → destination); configure separately for each direction if bi-directional needed

Azure File Share

  • Fully managed SMB (2.1, 3.0, 3.1.1) and NFS (4.1) file shares in the cloud
  • Port 445 must be open for SMB; often blocked by residential ISPs
  • Azure File Sync — sync on-prem Windows file servers with Azure Files; enables cloud tiering (infrequently accessed files stored in cloud, stubs on-prem)
  • Tiers: Transaction optimized, Hot, Cool (for files shares)
  • Auth: Storage account key, Entra ID (Kerberos for domain-joined), or SAS
  • Snapshots — point-in-time copies of file shares; can restore individual files

Storage Security

  • Storage Firewall — restrict to specific IPs or VNets; default allows all; "Trusted Microsoft services" exception for Azure backup, Event Grid
  • SSE (Storage Service Encryption) — AES-256 at rest; automatic; cannot disable; key options: Microsoft-managed, Customer-managed (CMK via Key Vault), Customer-provided
  • Infrastructure Encryption — double encryption (SSE + infrastructure layer); for highest compliance
  • Access Keys — two 512-bit keys per account; full access; rotate regularly
  • Shared Access Signature (SAS) — time-limited, permission-limited URI token
  • User Delegation SAS — signed with Entra ID credentials; most secure SAS type; doesn't use account key
  • Stored Access Policy — server-side policy; allows revocation without rotating account key
  • Entra ID RBAC — best practice; Storage Blob Data Contributor / Reader / Owner
  • Minimum TLS version — enforce TLS 1.2 minimum on storage accounts
SAS revocation: Compromised SAS with no Stored Access Policy → must rotate account key (affects ALL SAS tokens). With Stored Access Policy → delete policy to revoke only those SAS tokens.

AzCopy & Storage Explorer NEW

  • AzCopy — command-line tool for high-performance data transfer to/from Azure Storage and between storage accounts
  • azcopy copy 'C:\local\file' 'https://account.blob.core.windows.net/container?SAS'
  • azcopy sync — sync source to destination (like rsync); only copies changed files
  • azcopy copy --recursive — include subdirectories
  • Supports Blob, Files, Amazon S3 as source/destination
  • Authenticate with: SAS token, Entra ID (azcopy login), or managed identity
  • Azure Storage Explorer — GUI desktop app for managing storage accounts; supports all storage services; connect via account key, SAS, Entra ID, or Azure Stack
  • Azure Import/Export Service — ship physical hard drives to Azure datacenter for large data transfers (petabytes); use when network transfer would take too long
  • Azure Data Box — physical device shipped to you; load data offline; ship back to MS; for 40 TB – 100 TB transfers

Administer Azure Virtual Machines

08

Planning & VM Sizes

  • Families: General Purpose (D/B), Compute Optimized (F), Memory Optimized (E/M), Storage Optimized (L), GPU (N), HPC (H)
  • Resize — stop & deallocate first if moving to a different size family/cluster
  • VM disk types: OS disk (required), Temporary disk (NOT persistent — data lost on reboot/deallocate), Data disks (optional, persistent)
  • Spot VMs — unused Azure capacity at steep discount; can be evicted with 2-minute warning; use for non-critical batch/dev workloads

VM Storage (Disks)

  • Managed disks — Azure manages storage; recommended; supports snapshots, images, Zone replication
  • Disk types (perf/cost ascending): Standard HDD → Standard SSD → Premium SSD → Premium SSD v2 → Ultra Disk
  • Ultra Disk — sub-ms latency; configure IOPS/throughput independently; cannot be used as OS disk; requires Availability Zone
  • Ephemeral OS Disk — stored on VM host's local cache/temp storage; faster; no storage cost; lost on stop/deallocate; great for stateless VMs
  • Disk caching: None (best for write-heavy), ReadOnly (best for read-heavy data), ReadWrite (OS disk default)
  • Disk Bursting — Premium SSD / Standard SSD can burst IOPS beyond provisioned level temporarily (credit-based)
  • Shared Disks — attach one managed disk to multiple VMs simultaneously; for clustered workloads (Windows Server Failover Cluster)

Connecting to VMs

  • RDP (port 3389) — Windows VMs
  • SSH (port 22) — Linux VMs; use key pairs (public on VM, private on client)
  • Azure Bastion — fully managed PaaS; RDP/SSH via browser over HTTPS (443); no public IP on VM; no exposed 3389/22; Bastion deployed in AzureBastionSubnet (/27+)
  • Just-in-Time (JIT) VM Access — Defender for Cloud; opens port only for specified time/IP; logs all access
  • Serial Console — emergency access when network/OS is down; no network required
  • Run Command — run scripts on VM through Azure portal/CLI; no network access to VM needed

VM Extensions NEW

  • Small applications that provide post-deployment configuration and automation on Azure VMs
  • Custom Script Extension — run scripts (PowerShell/Bash) on a VM; scripts pulled from storage or GitHub; one-time execution; use for initial configuration
  • DSC Extension (Desired State Configuration) — Windows PowerShell DSC; enforce configuration state; idempotent; used with Azure Automation DSC
  • Azure Monitor Agent (AMA) — replaces legacy MMA/OMS agent; collects metrics/logs; configured via Data Collection Rules
  • Azure Diagnostics Extension — collect guest OS metrics/logs into Azure Monitor metrics or storage
  • BGInfo extension — displays system info on VM desktop background
  • Dependency Agent — used with VM Insights for service map (requires AMA)
  • Extensions can be deployed at VM creation or post-deployment; managed via portal, CLI, ARM, or policy
  • Extensions run with elevated privileges; verify source and integrity before deploying

High Availability

  • Availability Set — protects against hardware failures in same datacenter
  • Fault Domains (FD) — separate power/network racks; default 2, max 3
  • Update Domains (UD) — groups patched separately; default 5, max 20
  • SLA: 99.95% with 2+ VMs in availability set
  • Availability Zones — physically separate datacenters within a region; SLA 99.99%; requires Standard LB + Standard public IP
  • Regional Pairs — paired regions for geo-redundancy; updates to one region at a time
  • Proximity Placement Groups — ensure VMs are placed close together for ultra-low latency (within same datacenter/rack); good for HPC and tightly coupled apps
  • Dedicated Hosts — physical Azure server dedicated to your org; compliance/isolation; you control maintenance window

Virtual Machine Scale Sets (VMSS)

  • Deploy and manage a set of identical, auto-scaling VMs
  • Autoscaling — scale in/out based on metrics (CPU, memory, custom) or schedule
  • Supports up to 1000 VMs (custom image: 600)
  • Orchestration modes: Uniform (identical VMs) or Flexible (mix of sizes; manual + auto)
  • Scaling policies: min/max count, scale rules, cool-down period (default 5 min; prevents rapid scale in/out)
  • Scale-in policy: OldestVM (default), NewestVM, or Default (zone-balanced)
  • Integrates with Load Balancer or Application Gateway for traffic distribution

Azure Compute Gallery NEW

  • Centralized repository for storing, sharing, and versioning VM images and applications
  • Formerly called Shared Image Gallery
  • Image Definition — describes the image (OS, publisher, offer, SKU, state, VM generation)
  • Image Version — a specific version of the image; can be replicated to multiple regions
  • Share images across subscriptions, regions, and even tenants (RBAC-based sharing)
  • Supports generalized images (sysprepped/waagent deprovision) and specialized images (exact copy, retains machine info)
  • Use VM Image Builder — automated image creation pipeline; build on base images, customize with scripts, distribute to Compute Gallery
  • Replicate images to up to 50 regions; each region stores a specified replica count

Administer PaaS Compute Options

09

App Service Plans

  • Defines compute resources for App Service (location, size, instance count)
  • Tiers: Free → Shared → Basic → Standard → Premium → Isolated (ASE)
  • Free/Shared — shared infrastructure; no autoscale; no custom domain HTTPS
  • Basic — dedicated compute; custom domains/SSL; manual scale max 3 instances
  • Standard — autoscale, deployment slots (5), VNet integration, custom domain SSL
  • Premium — more slots (20), private endpoints, enhanced performance, VNet integration always-on
  • Isolated / App Service Environment (ASE) — dedicated VNet; highest isolation; for regulated/compliance workloads; own inbound/outbound IPs
  • Multiple apps can share one plan (share compute resources)

App Service Features

  • Custom domains — add CNAME or A record; verify via TXT record; enable HTTPS binding
  • Managed Identity — assign to App Service; access Key Vault/Storage without credentials
  • Authentication — built-in auth (Entra ID, Google, Facebook, Twitter, GitHub) — no code changes needed
  • VNet Integration — outbound calls to VNet resources (Standard+); not for inbound
  • Private Endpoint — inbound traffic via private IP (Premium/Isolated)
  • App Service Backup — requires Standard+; backup to storage account; max 10 GB per backup
  • WebJobs — background tasks within App Service; Continuous or Triggered
  • App Service Diagnostics — built-in troubleshooting wizard; no external tool needed
  • Autoscale — Standard+ tier; scale based on CPU/memory/HTTP queue length or schedule; not available on Basic

Deployment Slots

  • Separate live environments (staging, dev) within same App Service plan
  • Slot Swap — swap staging → production with zero downtime; warms up first; if swap fails, auto-rolls back
  • Traffic routing — send % of traffic to a slot (A/B testing)
  • Slot settings — connection strings or app settings marked "slot setting" stay with slot during swap
  • Standard = 5 slots; Premium = 20 slots

Azure Container Instances (ACI) & Container Apps

  • ACI — simplest container run; no VM or orchestrator; pay per second; fast startup; no auto-scale
  • Container Groups (ACI) — multi-container pod; share lifecycle, network, storage; similar to K8s pod
  • Container Apps — serverless containers on Kubernetes (KEDA); scale to zero; Dapr support; traffic splitting; managed platform (no K8s to manage)
  • ACI vs Container Apps: ACI = simple isolated tasks; Container Apps = microservices, event-driven, auto-scale

Azure Kubernetes Service (AKS) NEW

  • Managed Kubernetes cluster; Azure manages the control plane (free); you pay for agent (worker) nodes only
  • Node Pools — groups of VMs with same configuration; system node pool (required, runs K8s components) + user node pools (your workloads)
  • Autoscaler: Cluster Autoscaler (adds/removes nodes) and Horizontal Pod Autoscaler (adds/removes pods)
  • Networking modes:
    • Kubenet — pods get IPs from a separate space (NAT to VNet); simpler but limited
    • Azure CNI — pods get IPs directly from VNet subnet (can be reached directly); required for Windows nodes, VNet peering, private cluster
  • Private cluster — API server accessible only within VNet (no public endpoint); use Azure CNI
  • AKS + Azure Container Registry (ACR) — attach ACR to AKS using managed identity; AKS pulls images from ACR without credentials
  • Azure Monitor for Containers / Container Insights — monitor AKS node/pod metrics and logs in Log Analytics
  • Azure Policy for AKS — enforce Kubernetes admission policies using Azure Policy (e.g., no privileged containers)
  • Entra ID integration — Entra ID-based RBAC for cluster; users authenticate with Entra ID credentials
  • Upgrades — AKS supports in-place upgrades; new node image; use maintenance windows to control timing
Exam tip: AKS control plane is FREE. You pay for VMs (nodes), storage, and networking. Azure CNI = pods get VNet IPs directly. Kubenet = pods get NAT'd IPs from pod CIDR.

Administer Data Protection

10

Azure Backup

  • Cloud-based backup; stores backups in Recovery Services Vault
  • What can be backed up: Azure VMs, Azure Files, SQL in Azure VMs, SAP HANA, on-prem via MARS agent
  • Recovery Services Vault — stores backup data; choose LRS or GRS redundancy; create in same region as protected items
  • Backup Policy — frequency (daily/weekly) and retention (daily, weekly, monthly, yearly)
  • Azure VM backup — snapshot first (instant restore from snapshot, fast), then vault copy (longer retention)
  • Instant Restore — restore from snapshot (2–5 days retention); much faster than vault restore
  • MARS Agent — on-prem Windows file/folder/system state backup to vault
  • MABS / DPM — on-prem VMs + apps (SQL, Exchange, SharePoint)
  • Cross-region Restore (CRR) — restore Azure VM backups to secondary region from GRS vault

Backup Restore Options for Azure VMs

  • Create new VM — deploy a new VM from backup; original VM untouched
  • Replace existing disk — swap OS/data disk from backup onto existing VM
  • Restore disks — restore managed disks to storage; then attach manually
  • File Recovery — mount backup as a drive on a running VM; restore individual files; download a script, run it, get a mapped drive
  • Restore point (recovery point) types: Crash-consistent, App-consistent (VSS on Windows)

On-prem VMs + apps (SQL, Exchange)MABS or System Center DPM

Soft Delete & Vault Immutability

  • Soft Delete — deleted backup data retained for 14 additional days; protects accidental deletion; enabled by default for new vaults; incurs storage cost during retention
  • To permanently delete: disable soft delete, then delete (or wait 14 days)
  • Immutable Vault — vault-level immutability; prevents deleting/modifying backup data; locked immutability = cannot disable; stronger compliance
  • Multi-User Authorization (MUA) — require approval from another admin to disable soft delete or stop backups; extra protection against ransomware

Azure Site Recovery (ASR)

  • Disaster recovery — replicates VMs to secondary region/location; continuous replication
  • Replication: Azure VM → another Azure region; on-prem Hyper-V/VMware → Azure
  • RPO — max data loss; ASR achieves RPO as low as 30 seconds (app-consistent) or crash-consistent
  • RTO — time to recover; ASR targets minutes for Azure VMs
  • Failover: Test Failover (isolated test, no production impact), Planned Failover (no data loss), Unplanned Failover (some data loss possible)
  • Failback — return to primary site after recovery; re-protect first
  • Replication Policy — crash-consistent frequency (every 5 min), app-consistent snapshot frequency, retention
  • Uses Recovery Services Vault for ASR metadata
  • Recovery Plans — define failover sequence; add manual steps or Azure Automation runbooks; fail over multiple VMs in order

Administer Monitoring

11

Azure Monitor — Overview

  • Central monitoring platform — collects metrics, logs, and traces from all Azure resources
  • Metrics — numerical time-series; stored 93 days; near real-time; queryable via Metrics Explorer
  • Logs — structured records in Log Analytics Workspace; queryable via KQL; retention configurable (30 days–2 years)
  • Diagnostic Settings — push resource-level logs/metrics to: Log Analytics Workspace, Storage Account, Event Hub, or Partner solution
  • Sources: Azure resources (platform logs), Guest OS (VM extensions), Applications (App Insights), Custom (API)
  • Azure Monitor Workbooks — interactive reports combining queries, metrics, parameters, visualizations
  • VM Insights — pre-built monitoring for VMs (performance, processes, dependencies); requires AMA + Dependency Agent
  • Container Insights — pre-built monitoring for AKS and ACI; node/pod metrics, logs in Log Analytics

Azure Activity Log

  • Records ALL subscription-level operations — WHO did WHAT and WHEN on Azure resources
  • Captures: create/delete/modify resource, policy assignments, role assignments, service health events
  • Retained for 90 days by default; archive to Storage or Log Analytics for longer
  • For auditing and security investigations — "who deleted this resource?"
  • NOT for resource-level data (use Diagnostic Settings for that)
  • View in Azure portal → Monitor → Activity Log; filter by subscription, resource type, operation, time

Log Analytics Workspace & KQL

  • Central repository for log data from Azure Monitor, Activity Logs, VMs, containers
  • Data queried using KQL (Kusto Query Language) — pipe-based: Table | where | project | summarize | order by
  • Connect VMs with Azure Monitor Agent (AMA); configured via Data Collection Rules (DCR)
  • DCR specifies: what data to collect (performance counters, event logs) and where to send
  • One workspace can hold data from multiple subscriptions and regions

KQL Quick Reference

  • AzureActivity | where OperationNameValue contains "DELETE" | project TimeGenerated, Caller, ResourceGroup
  • Heartbeat | summarize LastSeen = max(TimeGenerated) by Computer | where LastSeen < ago(1h)
  • Perf | where ObjectName == "Processor" | summarize avg(CounterValue) by Computer, bin(TimeGenerated, 5m)
  • SecurityEvent | where EventID == 4625 | summarize count() by Account, Computer — failed logins
  • | take 10 — return N rows; | distinct Column — unique values; | extend newCol = expression

Azure Alerts

  • Alert Rule = Signal + Condition + Action Group
  • Signal types: Metric (threshold), Log (KQL query result), Activity Log, Resource Health, Smart Detection
  • Action Groups — reusable set of actions: Email/SMS, Webhook, Azure Function, Logic App, ITSM connector, push notification, Automation Runbook
  • Severity: 0 (Critical) → 1 (Error) → 2 (Warning) → 3 (Informational) → 4 (Verbose)
  • Alert States: New → Acknowledged → Closed
  • Stateful vs Stateless: Stateful alerts fire once when triggered, auto-resolve when condition clears. Stateless fire every evaluation if condition met.
  • Metric alerts: Static threshold or Dynamic threshold (ML-based baseline)

Application Insights NEW

  • APM (Application Performance Monitoring) service within Azure Monitor — for web apps and services
  • Collects: requests, dependencies, exceptions, page views, custom events, performance counters
  • Integration: add SDK to app code (Node.js, .NET, Java, Python) or use auto-instrumentation (App Service, AKS, Functions)
  • Live Metrics — real-time stream of telemetry (requests, failures, performance)
  • Application Map — visual map of app components and dependencies; shows failure rates and latency
  • Availability Tests — URL ping tests (from multiple regions) to check app is up; alerts on failure
  • Smart Detection — ML-based anomaly detection; alerts on unusual failure rates, performance degradation
  • Funnel / User Flows / Retention — usage analytics for understanding user behavior
  • Data stored in Log Analytics Workspace (workspace-based App Insights); query with KQL in same workspace

Azure Advisor NEW

  • Personalized cloud consultant — analyzes your Azure usage and recommends best practices
  • Five categories: Cost, Security, Reliability, Operational Excellence, Performance
  • Cost — right-size or shut down underutilized VMs, buy reservations, use Hybrid Benefit
  • Security — integrates with Microsoft Defender for Cloud; shows security posture recommendations
  • Reliability (formerly High Availability) — availability sets, zone redundancy, backup recommendations
  • Performance — Premium SSD recommendations, caching, CDN usage
  • Operational Excellence — resource health, service limits, ARM template best practices
  • Recommendations have an impact (High/Medium/Low) and estimated savings where applicable
  • Can configure Advisor Alerts — notify when new recommendations appear

Update Management NEW

  • Manage OS updates for Windows and Linux VMs (Azure + on-prem + other clouds)
  • Two versions: Update Management (classic) via Azure Automation, and newer Azure Update Manager (standalone service, recommended)
  • Azure Update Manager — no Log Analytics or Automation account needed; native Azure service
  • Assessment — evaluate missing updates on VMs; runs on demand or on schedule
  • Maintenance Configurations — schedule when updates are applied; define maintenance windows
  • Update Deployment — deploy specific updates to a set of VMs at scheduled time
  • Supports: Windows Server, Ubuntu, Red Hat, CentOS, SLES, Oracle Linux
  • Patch orchestration: Azure-managed patching (auto-patches); Customer-managed (you control schedule)
  • Hot Patching — apply updates to Windows Server without reboot (preview feature for supported VMs)
Last-minute tips: Activity Logs = who/what/when at subscription level (90 days). Diagnostic Settings = push resource logs to Log Analytics. Action Groups = reusable notification sets. App Insights = APM for code-level telemetry. Azure Advisor = 5 pillars: Cost, Security, Reliability, Ops Excellence, Performance.
